# ADR-004: Centralized Route Error Handler

**Status:** Active
**Area:** error-handling
**Date:** 2026-04-09

---

## Context

Every API route handler needs consistent error handling: logging the error, extracting a useful message, and returning a structured JSON response with an appropriate HTTP status code. Without a shared pattern, each route would implement its own try/catch with inconsistent response formats.

## Decision

All route handlers use `handleRouteError(c, error, context)` from `route-utils.ts` in their catch blocks. This function logs the error with the provided context string, attempts to extract an upstream HTTP status code (from errors that wrap service responses), and returns a `{ message }` JSON response. If the error has an upstream status code (400-599), that status is forwarded; otherwise, 500 is returned.

The alternative — Hono middleware-based error handling — is not used. Errors are caught per-handler with explicit context strings.

## Consequences

### Enables
- Consistent JSON error response format (`{ message }`) across all endpoints
- Error logging always includes a human-readable context string
- Upstream service errors (e.g., from Immich API) have their status codes preserved

### Costs
- Every route handler must include a try/catch block calling `handleRouteError` — boilerplate that could be eliminated with middleware
- The context string is a static message, not dynamically generated from the request
- The error type assertion (`error as Error & { response?: ... }`) is loose

---

## Rules

- **DO:** Wrap every route handler body in try/catch and call `handleRouteError` in the catch
- **DO:** Provide a descriptive context string that identifies the operation (e.g., "Failed to fetch images")
- **DON'T:** Return raw error messages to clients — always go through `handleRouteError`
- **DON'T:** Throw errors from route handlers without catching them
- **WHEN IN DOUBT:** Use `handleRouteError` even for simple handlers — consistency matters more than brevity

> **Scope:** All route files in `apps/server/src/routes/`

---

## Evidence

- `apps/server/src/routes/route-utils.ts`: single `handleRouteError` function definition
- Used in 12 of 14 route files (all except `index.ts` and `assets.ts` which has inline error handling)

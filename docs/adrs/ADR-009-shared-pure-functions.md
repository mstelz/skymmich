# ADR-009: Pure Functions in Shared Package for Business Logic

**Status:** Active
**Area:** shared
**Date:** 2026-04-09

---

## Context

Several business logic computations are needed by both the server (for data processing) and could potentially be needed by the client (for display or preview). These computations include image summary calculation, target grouping and enrichment, catalog CSV parsing, object name normalization, coordinate conversion, and visibility calculation. Embedding this logic in the server's storage layer would make it untestable without database setup and unavailable to the client.

## Decision

Domain computations are extracted as pure functions in `packages/shared/src/`. These functions take typed inputs and return typed outputs with no side effects — no database access, no service calls, no global state. They are unit tested in the same package (`*.test.ts` files alongside the source).

The server imports and calls these functions with data from the database. The shared package has no dependency on any server or client framework.

## Consequences

### Enables
- Unit testable without database or framework setup (`npm run test` runs shared tests directly)
- Reusable across server and client code via the `@shared` path alias
- Clear separation: data access in storage layer, computation in shared functions
- Functions serve as documentation of business rules (e.g., how integration hours are calculated)

### Costs
- Requires discipline to keep functions pure — it's tempting to add a "just one DB call" inside a shared function
- Input/output types must be defined in the shared package, which can lead to type duplication or adapter types

---

## Rules

- **DO:** Extract computations that don't need DB access into `packages/shared/src/`
- **DO:** Write pure functions — take inputs, return outputs, no side effects
- **DO:** Add unit tests for shared functions (co-located `*.test.ts` files)
- **DON'T:** Import from `apps/server/` or `apps/client/` in the shared package
- **DON'T:** Add database, HTTP, or framework dependencies to the shared package
- **WHEN IN DOUBT:** If a function needs data from the DB, keep it in the server and call the shared function with the data

> **Scope:** All files in `packages/shared/src/`

---

## Evidence

- `packages/shared/src/image-utils.ts`: `computeImageSummary()` and `groupAndEnrichTargets()` — pure functions with typed inputs
- `packages/shared/src/catalog-utils.ts`: `parseNGCCSV()`, `normalizeObjectName()`, coordinate conversion functions
- `packages/shared/src/visibility.ts`: `computeVisibility()` — pure astronomical calculation
- `packages/shared/src/image-utils.test.ts`, `catalog-utils.test.ts`, `visibility.test.ts`: unit tests

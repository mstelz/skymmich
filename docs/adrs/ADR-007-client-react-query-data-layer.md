# ADR-007: React Query as Data Layer with Manual Invalidation

**Status:** Active
**Area:** client
**Date:** 2026-04-09

---

## Context

The frontend needs a data fetching and caching layer for all API communication. The application is single-user, and data changes come from two sources: user actions (mutations via the REST API) and background events (plate solving, Immich sync via WebSocket). The default React Query behavior of auto-refetching on window focus and using short stale times can cause unnecessary network traffic.

## Decision

React Query is used for all data fetching. The global configuration sets `staleTime: Infinity`, `refetchOnWindowFocus: false`, `refetchInterval: false`, and `retry: false`. Data is only refreshed through explicit cache invalidation (`invalidateQueries`) after mutations, or via WebSocket-triggered refetches.

A custom `getQueryFn` function automatically builds query parameters from filter objects, using the API URL as the primary query key. The `apiRequest` helper provides a typed wrapper around `fetch` for mutations.

This configuration was not a deliberate architectural choice and may warrant revisiting — some reasonable auto-refetch behavior (e.g., on window focus after a long idle period) could improve data freshness.

## Consequences

### Enables
- No unnecessary background refetches — efficient for self-hosted deployments
- Predictable data flow: data only changes when the user takes an action or a WebSocket event fires
- Clean query key pattern: API URL + filter object makes cache management straightforward

### Costs
- If WebSocket disconnects, data can become stale with no automatic recovery
- Every mutation must explicitly invalidate affected query keys — forgetting this leaves stale data in the UI
- `retry: false` means transient network errors immediately surface as failures

---

## Rules

- **DO:** Use the API URL as the primary query key (e.g., `["/api/images"]`)
- **DO:** Invalidate relevant query keys after every mutation
- **DO:** Use WebSocket hooks to trigger refetches for background events
- **DON'T:** Manually call `fetch` for GET requests — use React Query
- **DON'T:** Rely on auto-refetch behavior — it is globally disabled
- **WHEN IN DOUBT:** Invalidate more query keys than fewer after a mutation — stale data is worse than an extra refetch

> **Scope:** `apps/client/src/lib/queryClient.ts` and all React Query usage in `apps/client/src/`

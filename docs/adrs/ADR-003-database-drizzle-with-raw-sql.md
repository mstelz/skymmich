# ADR-003: Drizzle ORM with Raw SQL Escape Hatches

**Status:** Active
**Area:** database
**Date:** 2026-04-09

---

## Context

Drizzle ORM provides a type-safe query builder, but some operations differ significantly between SQLite and PostgreSQL — particularly JSON array operations (SQLite stores arrays as JSON text, PostgreSQL has native arrays) and complex subqueries. A pure Drizzle abstraction for these would be verbose or impossible.

## Decision

Drizzle ORM is used for standard CRUD operations (select, insert, update, delete with conditions). For dialect-specific operations, raw SQL via Drizzle's `sql` template literal is used, often behind an `if (isPostgres)` branch. This includes tag array filtering, equipment subqueries, and catalog search.

Pure Drizzle is not abandoned — it is preferred for straightforward operations. Raw SQL is reserved for cases where the Drizzle API cannot express the query naturally for both dialects.

## Consequences

### Enables
- Type-safe queries for the common case
- Full access to dialect-specific SQL when needed
- Pragmatic approach: doesn't force awkward Drizzle patterns where raw SQL is clearer

### Costs
- Raw SQL fragments are not type-checked by Drizzle
- Dialect branches (`if (isPostgres)`) spread through the storage layer
- Risk of SQL injection if `sql` template literals are misused (though Drizzle's `sql` is parameterized by default)

---

## Rules

- **DO:** Use Drizzle's query builder for standard CRUD operations
- **DO:** Use `sql` template literals (parameterized) for dialect-specific operations
- **DON'T:** Use string concatenation to build SQL — always use Drizzle's `sql` template for safe parameterization
- **DON'T:** Add raw SQL where Drizzle's API can express the same query cleanly
- **WHEN IN DOUBT:** Write the Drizzle version first; fall back to raw SQL only if it doesn't work for both dialects

> **Scope:** `apps/server/src/services/storage.ts` and `apps/server/src/services/catalog.ts`

# ADR-006: Configuration Precedence — Env Vars over DB Settings over Defaults

**Status:** Active
**Area:** services
**Date:** 2026-04-09

---

## Context

Skymmich needs to be configurable through multiple channels: environment variables (for Docker/container deployments), a database-stored admin UI (for runtime changes without redeployment), and sensible defaults (for zero-config startup). These sources can conflict, so a clear precedence is needed.

## Decision

Configuration follows a three-tier precedence: environment variables override database-stored admin settings, which override hardcoded defaults. This is implemented in `ConfigService.getConfig()` which merges all three layers.

Environment variable overrides are limited to path-related settings (`IMMICH_MAPPING_PATH`, `LOCAL_MAPPING_PATH`, `XMP_SIDECAR_PATH`) where container volume mounts make environment variables the natural configuration mechanism. All other settings are managed through the admin UI.

## Consequences

### Enables
- Docker deployments can set path mappings via environment variables without touching the admin UI
- Users can configure everything else through the web-based admin panel
- First-run experience works with empty defaults — features are disabled until configured

### Costs
- Environment variable overrides are invisible in the admin UI — a user might change a path in the UI and wonder why it doesn't take effect
- The set of env-var-overridable settings is hardcoded, not discoverable
- Config is cached in memory after first load; changes via env vars require a restart

---

## Rules

- **DO:** Use the admin UI for all non-path settings (API keys, feature flags, sync options)
- **DO:** Use environment variables only for container path mappings
- **DON'T:** Add new environment variable overrides without clear justification (container deployment need)
- **DON'T:** Read `process.env` directly in service code — go through `ConfigService`
- **WHEN IN DOUBT:** Make the setting configurable via admin UI, not environment variable

> **Scope:** `apps/server/src/services/config.ts` and `.env.example`

# ADR-001: Dual Database Schema with Manual Sync

**Status:** Active
**Area:** database
**Date:** 2026-04-09

---

## Context

Skymmich was originally built with SQLite as its only database. PostgreSQL support was added later to accommodate users who prefer a dedicated database server. Drizzle ORM requires separate schema definitions for each database dialect (SQLite uses `sqliteTable`, PostgreSQL uses `pgTable`), so a single unified schema file was not an obvious option. Whether Drizzle supports a dialect-agnostic schema definition has not been investigated.

## Decision

Two separate schema files are maintained in parallel: `packages/shared/src/db/pg-schema.ts` and `packages/shared/src/db/sqlite-schema.ts`. Both files define the same tables and columns but use dialect-specific Drizzle constructors. A comment at the top of each file warns that they must be kept in sync. At runtime, the `isPostgres` flag (set based on the presence of `DATABASE_URL`) selects which schema and driver to use.

The alternative — a single schema targeting both dialects — was not used because it was the quickest path to add PostgreSQL support on top of the existing SQLite implementation.

## Consequences

### Enables
- SQLite works out of the box with zero configuration
- PostgreSQL is available for users who need it (e.g., existing database infrastructure, concurrent access)
- Each schema can use dialect-native types and features

### Costs
- Every column addition or change must be made in both files — easy to forget, and there is no automated check
- Several places in the codebase use `isPostgres` conditionals for dialect-specific SQL (JSON array operations, tag filtering), increasing complexity
- Migration tooling differs: SQLite uses Drizzle Kit migrations auto-applied on startup; PostgreSQL uses a separate manual migration script

---

## Rules

- **DO:** Update both `pg-schema.ts` and `sqlite-schema.ts` when adding, removing, or modifying any column or table
- **DO:** Test changes against both SQLite and PostgreSQL when modifying schema or query logic
- **DON'T:** Add dialect-specific features to one schema without considering the other
- **WHEN IN DOUBT:** Favor SQLite compatibility first, since it is the default and most-used path

> **Scope:** All schema files in `packages/shared/src/db/` and any code that reads `isPostgres`

# SPEC-014: Admin Panel & Configuration

**Status:** Active
**Area:** system
**Date:** 2026-03-23

---

## Purpose

This feature provides a centralized settings interface for configuring all integrations and application behavior. It manages Immich connection settings, Astrometry.net configuration, XMP sidecar options, and application-level flags. It also exposes system monitoring (cron jobs, worker status, database info).

---

## Behavior

The admin page presents categorized settings forms. Configuration is stored in the database and loaded into a cached singleton at runtime. Environment variables can override specific path-related settings.

### Entry Points

- `/admin` — Admin settings page
- `POST /api/admin/settings` — Save admin settings
- `GET /api/admin/settings` — Get admin settings (API keys redacted)
- `POST /api/test-immich-connection` — Test Immich connectivity
- `POST /api/test-astrometry-connection` — Test astrometry.net API key
- `GET /api/admin/cron-jobs` — Get cron job status
- `GET /api/admin/database` — Get database type, size, and path info
- `GET /api/admin/database/backup` — Download SQLite database backup
- `GET /api/health` — Health check endpoint

### Key Flows

**Settings management:**
1. Admin page loads current settings (with redacted API keys)
2. User modifies settings and saves
3. Server preserves existing API keys if the submitted value is redacted (starts with `••••`)
4. Settings are validated (Immich URL protocol check, etc.)
5. Config cache is cleared, forcing reload on next access
6. Cron jobs are reconfigured if sync frequency changed

**Connection testing:**
1. Immich: validates URL format, then attempts to fetch albums from the server (10-second timeout)
2. Astrometry: attempts to authenticate with the provided API key via the astrometry.net login endpoint

**Health check:**
1. Checks database connectivity by running a stats query
2. Reports worker status (enabled, running, PID, restart attempts)
3. Returns overall health status with uptime, versions, and timestamps

## Patterns Used

- `ConfigService` singleton with lazy-loaded, cached configuration
- Database-stored settings with hardcoded defaults as fallback
- API key redaction on read, preservation on write (redacted keys are not saved)
- Environment variable overrides for path settings (`IMMICH_MAPPING_PATH`, `LOCAL_MAPPING_PATH`, `XMP_SIDECAR_PATH`)

---

## Business Rules

- **API key redaction:** API keys are masked to `••••{last4}` when returned to the client. If a redacted value is submitted back, the server keeps the existing stored key.
- **URL protocol validation:** Immich host URLs must use HTTP or HTTPS. Other protocols are rejected to prevent SSRF.
- **Config precedence:** Environment variables > admin settings > hardcoded defaults.
- **SQLite backup only:** Database backup download is only available for SQLite databases. PostgreSQL users are directed to use `pg_dump`.
- **Health endpoint semantics:** Returns 200 if database is healthy, 503 if unhealthy. Designed for container orchestration health probes.

---

## Edge Cases & Constraints

- Config is cached in memory after first load — changes require clearing the cache (handled automatically by `updateConfig`)
- Connection tests use a 10-second timeout to avoid blocking the UI
- The health endpoint catches its own errors to always return a structured response

---

## Known Limitations

- No authentication or authorization — the admin panel is accessible to anyone with network access to the application
- No audit log of settings changes
- No validation of Immich path mapping correctness (user must ensure paths match between containers)
- Cron job reconfiguration after settings change may not take effect until the next cron cycle

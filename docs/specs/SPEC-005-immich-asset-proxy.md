# SPEC-005: Immich Asset Proxy

**Status:** Active
**Area:** immich-integration
**Date:** 2025-07-12

---

## Purpose

This feature proxies image requests from the Skymmich frontend to the Immich server. It keeps the Immich API key server-side so it is never exposed to the browser, and provides a consistent URL scheme for image thumbnails and originals regardless of the Immich server's actual location.

---

## Behavior

The proxy intercepts requests to `/api/assets/:assetId/:type` and forwards them to the configured Immich server with the stored API key, streaming the response back to the client.

### Entry Points

- `GET /api/assets/:assetId/:type` — Proxy an Immich asset (thumbnail, original, or fullsize)

### Key Flows

1. Client requests an image via `/api/assets/{immichId}/thumbnail` (or other type)
2. Server validates the asset ID format (alphanumeric with hyphens) and the type (must be one of: `thumbnail`, `original`, `fullsize`)
3. Server validates the configured Immich URL protocol (HTTP/HTTPS only) to prevent SSRF
4. Server forwards the request to `{immichHost}/api/assets/{assetId}/{type}` with the Immich API key in the `X-API-Key` header
5. Query parameters (e.g., `?size=preview`) are forwarded as-is
6. The response body is streamed back to the client with the original content-type header

---

## Patterns Used

- Server-side API key isolation — the Immich API key never reaches the client
- Request proxying with input validation
- Streaming response (response body passed through without buffering)

---

## Business Rules

- **Allowed asset types:** Only `thumbnail`, `original`, and `fullsize` are accepted. Other values return 400.
- **Asset ID format:** Must match `/^[a-zA-Z0-9-]+$/`. Invalid formats return 400.
- **SSRF protection:** The Immich host URL protocol is validated to only allow `http:` or `https:`. Invalid protocols return 400.
- **Missing configuration:** If Immich host or API key is not configured, returns 503.

---

## Edge Cases & Constraints

- If the Immich server returns a non-OK response, the proxy forwards the status code and status text
- The proxy only forwards `content-type` from the upstream response; other headers are stripped

---

## Known Limitations

- No caching layer — every request is forwarded to Immich, even for the same thumbnail
- No rate limiting on proxy requests
- Error responses from Immich are returned as JSON `{ message: statusText }` rather than forwarding the original Immich error body

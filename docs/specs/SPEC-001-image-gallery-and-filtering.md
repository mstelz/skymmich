# SPEC-001: Image Gallery & Filtering

**Status:** Active
**Area:** image-management
**Date:** 2025-07-10

---

## Purpose

This feature provides the primary interface for browsing and discovering astrophotography images. Users view their library as a filterable grid, drill into individual images, and track high-level collection statistics. It serves as the home page and main entry point for all image-related workflows.

---

## Behavior

The home page displays a responsive card grid of astrophotography images. Each card shows a thumbnail, capture date, plate-solving status badge, exposure metadata (focal length, aperture, ISO, exposure time), and integration summary (frame count, total hours). A stats bar displays aggregate collection metrics.

### Entry Points

- `GET /` — Home page (SPA route)
- `GET /api/images` — Fetch images with optional server-side filters
- `GET /api/stats` — Collection statistics
- `GET /api/tags` — Popular tags with counts
- `GET /api/constellations` — Distinct constellation values from image data

### Key Flows

1. On page load, the client fetches images, stats, tags, and equipment in parallel via React Query
2. Server-side filters (object type, tags, plate-solved status, constellation, equipment ID) are applied as query parameters to `/api/images`
3. Client-side filters (text search on title, date range, minimum integration hours) further narrow the result set
4. Images display in a paginated grid — 12 initially, with "Load More" to reveal 12 more at a time
5. Clicking an image opens the overlay view and updates the URL to `/?image={id}` for bookmarkability
6. Deep-linking: loading a URL with `?image={id}` auto-opens that image's overlay once data loads
7. Closing the overlay removes the `image` parameter from the URL
8. A sidebar displays popular tags; clicking a tag toggles it as a filter
9. Active filters display as removable badges with a "Clear All" option
10. WebSocket events for plate-solving completion and Immich sync trigger automatic data refresh

---

## Patterns Used

- React Query for data fetching with automatic cache invalidation
- WebSocket hooks (`usePlateSolvingUpdates`, `useImmichSyncUpdates`) for real-time refresh
- URL-driven state via `wouter` for deep-linking
- Server-side + client-side hybrid filtering: structured filters go to the API, free-text and date filters are applied client-side

---

## Business Rules

- **Tag relevance filtering:** Tags displayed in the sidebar and used for filtering exclude individual star names from plate-solving machine tags. Only catalog objects (NGC, IC, Messier, etc.), named nebulae/clusters/galaxies, and user-created tags are shown.
- **Filter combination:** All active filters are AND-combined. Multiple tags are OR-combined (image must have at least one matching tag).
- **Equipment filter uses subquery:** Filtering by equipment ID uses a SQL subquery on the image_equipment join table rather than loading all relationships into memory.

---

## Edge Cases & Constraints

- All images are loaded from the API at once; client-side pagination only controls rendering. This is a known pragmatic shortcut — at scale (hundreds+ images), the full dataset transfer could become a performance concern.
- Deep-link to a non-existent image ID silently does nothing (no error shown).
- Tag-based filtering on SQLite uses `json_each()` to search JSON-stored arrays; PostgreSQL uses native array overlap (`&&`).

---

## Known Limitations

- No server-side pagination — the API returns all matching images in one response
- Text search is client-side only (title substring match), not full-text search
- No sort controls — images display in database insertion order
- The equipment gallery section at the bottom of the home page is not filterable independently

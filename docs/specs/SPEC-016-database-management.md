# SPEC-016: Database Management

**Status:** Active
**Area:** system
**Date:** 2026-03-23

---

## Purpose

This feature manages the application's database lifecycle — supporting both SQLite (default) and PostgreSQL, running migrations automatically on startup, providing database info and backup capabilities, and exposing a health check endpoint for container orchestration.

---

## Behavior

The database engine is selected based on the presence of the `DATABASE_URL` environment variable. SQLite migrations are auto-applied on startup. Admin endpoints expose database diagnostics and backup.

### Entry Points

- `GET /api/admin/database` — Database type, path, size, and last modified timestamp
- `GET /api/admin/database/backup` — Download SQLite database file as attachment
- `GET /api/health` — Health check (database connectivity + worker status)
- Startup: auto-migration for SQLite databases

### Key Flows

**Database selection:**
1. If `DATABASE_URL` is set, connect to PostgreSQL using the connection string
2. If not set, use SQLite with a local file (`local.db` by default)
3. The `isPostgres` flag is set at module level and used throughout the codebase for dialect-specific SQL

**Auto-migration (SQLite):**
1. On startup, Drizzle Kit migrations from `tools/migrations/sqlite/` are applied automatically
2. Migration journal tracks which migrations have been applied

**Database info:**
1. Returns database type (sqlite/postgresql)
2. For SQLite: includes file path, size in bytes, and last modified timestamp

**Backup:**
1. Only available for SQLite
2. Reads the entire database file into memory and returns it as a downloadable attachment
3. Filename includes a timestamp: `skymmich-backup-{timestamp}.db`

## Patterns Used

- Dual-schema architecture: `pg-schema.ts` and `sqlite-schema.ts` must be kept in sync manually
- Runtime database selection via environment variable
- `isPostgres` flag for conditional SQL dialect (JSON operations, array handling, etc.)
- Drizzle ORM with dialect-specific drivers (`better-sqlite3` for SQLite, `pg` for PostgreSQL)

---

## Business Rules

- **SQLite is the default:** No configuration needed — the app works out of the box with SQLite.
- **Schema parity:** Both database schemas must be kept in sync. A comment at the top of each schema file warns about this.
- **Backup is SQLite-only:** PostgreSQL users are expected to use `pg_dump` or other external tools.
- **Health check database probe:** The health endpoint tests database connectivity by running a stats query, not a simple ping.

---

## Edge Cases & Constraints

- SQLite backup reads the entire file into memory — large databases could cause memory pressure
- The backup endpoint does not use SQLite's online backup API, so it reads the file as-is (potential for partial reads under concurrent writes, though SQLite's WAL mode mitigates this)
- PostgreSQL migrations are handled separately via `tools/scripts/apply-pg-migrations.ts`

---

## Known Limitations

- No automatic PostgreSQL migration on startup — only SQLite auto-migrates
- No database migration tool for moving data between SQLite and PostgreSQL (a manual `migrate-db.js` script exists but requires setup)
- Dual-schema maintenance is error-prone — there is no automated check that the schemas are in sync
- No database connection pooling configuration exposed to users

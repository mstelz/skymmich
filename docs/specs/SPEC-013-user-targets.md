# SPEC-013: User Targets & Observation Planning

**Status:** Active
**Area:** sky-and-targets
**Date:** 2026-03-20

---

## Purpose

This feature allows users to curate a personal list of observation targets — deep-sky objects they want to image or are actively tracking. Users annotate catalog objects with notes and tags to organize their planning. The targets page also shows a grouped view of already-imaged targets with aggregated statistics.

---

## Behavior

User targets are annotations layered on top of catalog objects. The targets page combines these user annotations with a grouped view of images by target name, enriched with catalog metadata.

### Entry Points

- `/targets` — Targets page
- `GET /api/user-targets` — List all user target annotations
- `GET /api/user-targets/:catalogName` — Get a single user target
- `PUT /api/user-targets/:catalogName` — Upsert a user target annotation (create or update)
- `DELETE /api/user-targets/:catalogName` — Delete a user target
- `GET /api/targets` — Get grouped image targets with catalog enrichment (computed server-side)

### Key Flows

**Target grouping (imaged targets):**
1. Fetch all images from the database
2. Group images by `targetName` (falling back to `title` if no target name)
3. For each group: count images, sum integration hours, get latest capture date, pick most recent thumbnail
4. Enrich with catalog data: look up each target name in the catalog for object type, constellation, magnitude, and common names
5. Sort by image count descending, then by latest capture date

**User target annotations:**
1. User selects a catalog object and adds notes and/or tags
2. Upsert semantics: PUT creates or updates based on `catalogName`
3. Tags are freeform strings — can be used for priority ("high-priority"), status ("in-progress"), or any user-defined categorization

**Visibility computation:**
1. Given an object's RA/Dec and the observer's latitude, compute:
   - Best observation month (when the object transits at midnight)
   - Prime season: +/- 3 months around best month
   - Status: circumpolar, seasonal, or never-rises
   - Maximum altitude at meridian transit

## Patterns Used

- Pure function for target grouping and enrichment (`groupAndEnrichTargets` in shared package)
- Pure function for visibility computation (`computeVisibility` in shared package)
- Upsert semantics for user target annotations (keyed by catalog name)
- Catalog enrichment: image data is joined with catalog metadata at query time

---

## Business Rules

- **Target name sources:** Target names come from plate-solving auto-matching, manual assignment via the target picker, or the image title as fallback.
- **Grouping key:** Images are grouped by `targetName` (or `title` if null). This means untitled images without target names each form their own group.
- **Catalog enrichment priority:** Catalog data (type, constellation, magnitude) overrides image-level values when available.
- **Visibility is observer-dependent:** Visibility calculations require a latitude. Without one, visibility data is not computed.

---

## Edge Cases & Constraints

- User targets reference catalog objects by name string, not by foreign key — if the catalog reloads with different name normalization, references could break
- Images with the same title but no target name will be grouped together, which may not be desired
- Visibility computation uses a simplified month-based heuristic, not precise ephemeris calculations

---

## Known Limitations

- No dedicated priority or status fields — these are handled via freeform tags
- No integration with external observation planning tools or almanac data
- No notification when a target enters its prime observation season
- Visibility "best month" is approximate (month granularity, not date-specific)

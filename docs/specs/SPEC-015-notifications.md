# SPEC-015: Notifications

**Status:** Active
**Area:** system
**Date:** 2026-02-25

---

## Purpose

This feature provides a system notification feed that surfaces events from background operations — Immich sync completions, plate-solving results, errors, and other system events. Notifications help users stay informed about asynchronous operations without polling the UI.

---

## Behavior

Notifications are created by backend services and stored in the database. The frontend polls for notifications and receives real-time updates via WebSocket when notifications change.

### Entry Points

- `GET /api/notifications` — Fetch all notifications
- `POST /api/notifications/:id/acknowledge` — Acknowledge a single notification
- `POST /api/notifications/acknowledge-all` — Acknowledge all notifications
- WebSocket event: `notifications-updated` — broadcast when notifications change

### Key Flows

1. Backend services create notification records in the database (e.g., sync completed, plate solving failed)
2. Frontend fetches notifications on load and subscribes to WebSocket updates
3. User can acknowledge individual notifications or acknowledge all at once
4. Acknowledging broadcasts a `notifications-updated` WebSocket event to refresh other connected clients
5. A daily cron job cleans up old notifications

---

## Patterns Used

- Database-stored notifications with WebSocket broadcast on changes
- Cron-scheduled cleanup for old notifications
- Acknowledge semantics (mark as read) rather than delete

---

## Business Rules

- **Acknowledge is idempotent:** Acknowledging an already-acknowledged notification is not an error.
- **Automatic cleanup:** Old notifications are cleaned up daily by the cron manager to prevent unbounded growth.
- **Real-time broadcast:** Any notification state change broadcasts to all connected WebSocket clients.

---

## Edge Cases & Constraints

- Notifications are not user-scoped — all notifications are visible to all users (single-user assumption)
- The cleanup schedule and retention period are managed internally by the cron manager

---

## Known Limitations

- No notification categories or severity levels in the UI
- No persistent unread count badge in the header
- No email or push notification — in-app only
- No filtering or search within notifications

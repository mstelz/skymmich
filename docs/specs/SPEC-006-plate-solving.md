# SPEC-006: Astrometry.net Plate Solving

**Status:** Active
**Area:** plate-solving
**Date:** 2025-07-12

---

## Purpose

This feature determines the precise sky coordinates of astrophotography images by submitting them to the Astrometry.net plate-solving service. Once solved, images are enriched with RA/Dec coordinates, pixel scale, field of view, rotation angle, constellation, catalog object tags, and a target name. This data powers the sky map, annotation overlay, and target grouping features.

---

## Behavior

Plate solving can be triggered manually (single or bulk) or automatically via a background worker. The process downloads the image from Immich, uploads it to astrometry.net, polls for results, and updates the image record with the solution.

### Entry Points

- `POST /api/plate-solving/images/:id/plate-solve` — Solve a single image (synchronous — waits for completion)
- `POST /api/plate-solving/bulk` — Submit multiple images for solving (asynchronous — returns submission IDs)
- `POST /api/plate-solving/update/:jobId` — Check and update a job's status
- `GET /api/plate-solving/jobs` — List all plate-solving jobs
- Background worker: `PlateSolvingWorker` (runs in production, checks for unsolved images on a configurable interval)

### Key Flows

**Manual single solve:**
1. Validate that plate solving is enabled and API key is configured
2. Download image from Immich using the stored fullUrl and Immich API key
3. Authenticate with astrometry.net (`/api/login`)
4. Upload image as multipart form data (`/api/upload`)
5. Create a `PlateSolvingJob` record with status "processing"
6. Broadcast WebSocket update
7. Poll submission status until calibration data appears (configurable interval, max ~60 min)
8. Fetch complete results: calibration (RA, Dec, pixel scale, orientation), annotations (object positions), and machine tags
9. Update the image record: set `plateSolved: true`, populate coordinate fields, determine constellation from coordinates, merge relevant machine tags, auto-match target name from catalog
10. Write XMP sidecar file if sidecar generation is enabled
11. Broadcast WebSocket completion update

**Background worker:**
1. Worker process runs as a separate Node.js child process (spawned by WorkerManager in production)
2. Connects to the main server via WebSocket for real-time status updates
3. Periodically scans for unsolved images and submits them
4. Checks processing jobs for completion
5. Configurable concurrency limit (default: 3 concurrent jobs)
6. Exponential backoff reconnection if WebSocket drops

## Patterns Used

- Long-polling with configurable interval for astrometry.net result checking
- Separate worker process with WebSocket communication to main server
- WorkerManager with supervised process lifecycle (restart on crash, exponential backoff, max attempts)
- Machine tag filtering: star names are excluded, only catalog objects kept

## Business Rules

- **Requires Immich for image access:** The plate-solving workflow downloads images via the Immich API. Images without an Immich-accessible URL cannot be plate-solved.
- **Configuration gating:** Plate solving requires both `enabled: true` and a valid `apiKey` in admin settings. The auto-worker additionally requires `autoEnabled: true`.
- **Constellation detection:** After solving, the image's constellation is determined from RA/Dec using simplified IAU boundary data.
- **Target auto-matching:** Plate-solving tags are matched against the OpenNGC catalog to automatically set a `targetName`. Messier objects are prioritized over NGC/IC; among same-priority matches, brightest (lowest V-mag) wins.
- **Tag deduplication:** Machine tags from astrometry.net are merged with existing image tags, filtered for relevance, and deduplicated.
- **Bulk submit is fire-and-forget:** Bulk submission returns immediately with submission IDs; completion is tracked via job status polling or WebSocket events.

---

## Edge Cases & Constraints

- Astrometry.net jobs can expire after ~30 days — checking an expired job returns a 404 and is marked as failed
- If astrometry.net reports `processing_finished` with no calibration data, the job is marked as failed
- The worker uses exponential backoff for WebSocket reconnection (1s to 30s max)
- Image download from Immich has a 30-second timeout; upload to astrometry.net has a 60-second timeout

---

## Known Limitations

- Plate solving only works for images accessible via the Immich API — no local file path support
- No hints (estimated RA/Dec, scale) are sent to astrometry.net, which can slow solving for wide-field images
- The synchronous single-solve endpoint blocks the request until completion (can be minutes)
- XMP sidecar writing may fail silently in Docker environments due to volume/permission issues

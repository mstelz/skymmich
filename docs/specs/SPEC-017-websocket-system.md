# SPEC-017: Real-Time WebSocket System

**Status:** Active
**Area:** infrastructure
**Date:** 2025-07-12

---

## Purpose

This feature provides real-time communication between the server and connected clients. It enables live updates for plate-solving progress, Immich sync completion, and notification changes without requiring the client to poll.

---

## Behavior

A WebSocket server runs alongside the HTTP server. Clients connect on page load and receive broadcast events. The background worker also connects via WebSocket to relay its status updates.

### Entry Points

- `ws://host/ws` — WebSocket endpoint (server-side: `WsManager`)
- Client-side: `WebSocketManager` singleton with React hooks

### Key Flows

**Server-side:**
1. `WsManager` creates a `WebSocketServer` attached to the Node HTTP server on the `/ws` path
2. Tracks connected clients in a `Set`
3. `broadcast(event, data)` sends JSON-encoded `{ event, data }` to all connected clients with `OPEN` readyState
4. Services call `wsManager.broadcast()` to push events (plate-solving updates, sync completion, notification changes)

**Client-side:**
1. `WebSocketManager` singleton connects on first hook usage
2. Ref-counted: connection lives as long as at least one component uses a socket hook
3. Incoming messages are parsed as `{ event, data }` and dispatched to registered event handlers
4. Exponential backoff reconnection on disconnect (1s to 30s max)
5. React hooks: `useSocket()` for raw access, `usePlateSolvingUpdates(callback)` and `useImmichSyncUpdates(callback)` for typed event subscriptions

**Worker connection:**
1. The `PlateSolvingWorker` process connects to the main server's WebSocket as a client
2. Sends status updates (job progress, completion) via the same `{ event, data }` protocol
3. Independent exponential backoff reconnection

---

## Patterns Used

- Singleton WebSocket manager (server and client)
- Broadcast-only pattern (server pushes to clients; no client-to-server commands)
- Ref-counted client singleton: connection is created on first hook mount, destroyed when last hook unmounts
- Exponential backoff with cap for reconnection on both server and client sides

---

## Business Rules

- **Broadcast-only:** The WebSocket is used for server-to-client push only. Clients do not send commands via WebSocket — all mutations go through REST API.
- **Protocol auto-detection:** Client detects `wss:` vs `ws:` based on the page's `location.protocol`.
- **No authentication:** WebSocket connections are not authenticated.

---

## Edge Cases & Constraints

- If the WebSocket connection drops, events during the disconnect window are lost (no replay/queue)
- The worker process connects to `ws://localhost:5000/ws` — this hardcoded URL assumes the worker and server are co-located
- Dead clients (connected but not responsive) are only removed on `close` or `error` events, not via heartbeat

---

## Known Limitations

- No message queuing — events during disconnection are permanently lost
- No heartbeat/ping mechanism to detect stale connections
- No event namespacing or versioning
- Worker WebSocket URL is hardcoded to localhost:5000
